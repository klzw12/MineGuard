# MineGuard 后端模块拆解 - 服务模块

> 本文档用于描述"服务模块（可执行应用）"的职责与交互关系，属于规划/目标结构文档。  
> YAML 配置拆分、加载优先级与"避免重复导致错乱"的规则，统一以 `docs/后端拆解-yml-rule.md` 为准（服务模块是唯一允许持有 `application*.yml(yaml)` 的位置）。

## 1. user-service 用户服务

### 1.1 模块职责

- 用户管理：用户注册、登录、信息修改、密码重置
- 认证授权：JWT Token 生成和验证、权限控制
- 角色管理：角色定义、角色分配、权限配置
- 司机管理：司机信息管理、状态管理
- 用户信息查询：用户详情查询、用户列表查询
- 用户状态管理：用户启用/禁用、状态变更

### 1.2 模块依赖关系

#### 模块间依赖

- **依赖**：所有公共配置模块
- **被依赖**：其他业务服务模块（通过 HTTP Exchange Client 调用）
- **Client 接口**：提供 UserClient、DriverClient 接口供其他服务调用

## 10. 服务调用关系

### 10.1 服务间调用机制

本项目服务间调用采用 **Spring 6 HTTP Exchange** 机制，而非传统的 FeignClient。

**所有 Client 接口定义在 `mineguard-common-core` 模块中**，使用 `@HttpExchange` 注解：

```java
@HttpExchange
public interface UserClient {
    @GetExchange("/user/exists/{id}")
    Result<Boolean> existsUser(@PathVariable("id") Long id);

    @GetExchange("/user/{id}")
    Result<Object> getUserById(@PathVariable("id") Long id);
}
```

### 10.2 Client 接口列表

| Client 接口 | 说明 | 所在模块 |
| --- | --- | --- |
| UserClient | 用户服务调用 | mineguard-common-core |
| VehicleClient | 车辆服务调用 | mineguard-common-core |
| TripClient | 行程服务调用 | mineguard-common-core |
| DispatchClient | 调度服务调用 | mineguard-common-core |
| WarningClient | 预警服务调用 | mineguard-common-core |
| CostClient | 成本服务调用 | mineguard-common-core |
| AiClient | AI服务调用 | mineguard-common-core |
| StatisticsClient | 统计服务调用 | mineguard-common-core |
| DriverClient | 司机服务调用 | mineguard-common-core |
| MessageClient | 消息服务调用 | mineguard-common-core |
| TaskClient | 任务服务调用 | mineguard-common-core |
| TransportClient | 运输服务调用 | mineguard-common-core |
| PythonClient | Python服务调用 | mineguard-common-core |
| GaodeMapClient | 高德地图API调用 | mineguard-common-map |

### 10.3 服务依赖关系矩阵

| 服务模块 | 调用的 Client | 被调用 |
| --- | --- | --- |
| user-service | 无 | UserClient, DriverClient |
| trip-service | UserClient, VehicleClient, DispatchClient | TripClient |
| dispatch-service | UserClient, VehicleClient, DriverClient | DispatchClient |
| warning-service | UserClient, VehicleClient, TripClient | WarningClient |
| statistics-service | UserClient, VehicleClient, TripClient, WarningClient | StatisticsClient |
| cost-service | UserClient, VehicleClient, TripClient | CostClient |
| ai-service | UserClient, VehicleClient, TripClient, WarningClient, StatisticsClient | AiClient |
| python-service | 无 | PythonClient |
| gateway-service | 无（路由层） | 无 |

**重要说明**：
- 服务间调用通过 Client 接口实现，而非直接依赖其他服务模块
- Client 接口统一放在 `mineguard-common-core` 中，避免循环依赖
- 服务注册发现通过 Nacos 实现，Client 通过服务名调用目标服务

### 10.4 服务端口分配

| 服务名称 | 端口 | 说明 |
| --- | --- | --- |
| user-service | 8081 | 用户服务 |
| trip-service | 8084 | 行程服务 |
| gateway-service | - | 网关服务 |

### 10.5 服务依赖层次

```tree

├── trip-service (行程管理)

Level 4: 增强服务
```

## 11. 服务间通信

### 11.1 HTTP Exchange 调用

- **调用方式**：Spring 6 声明式 HTTP 客户端
- **负载均衡**：基于 Nacos 服务发现
- **熔断降级**：Resilience4j 熔断器
- **超时控制**：统一超时配置

### 11.2 消息队列

- **消息生产者**：trip-service、warning-service、dispatch-service
- **消息消费者**：warning-service、statistics-service、ai-service
- **消息类型**：车辆状态变更、行程状态更新、预警触发
- **消息可靠性**：事务消息、消息重试、死信队列

### 11.3 WebSocket

- **实时推送**：user-service、trip-service
- **推送对象**：调度中心、司机端、前端监控页面
- **消息类型**：车辆状态变更、预警通知、调度指令、系统公告

## 14. 服务版本管理

服务模块同样遵循父 POM 统一版本策略：

- **坐标统一**：`groupId` 与项目父 POM 保持一致（当前为 `com.klzw`）。
- **版本统一**：使用父 POM 的 `${mineguard.version}`（当前为 `0.0.1-SNAPSHOT`）。
- **依赖不写 version**：由父 POM 的 `<dependencyManagement>` 统一锁定，避免多处维护导致版本漂移。

详细版本策略与依赖管理，请参考 `docs/后端拆解-依赖关系和版本文档.md`。

## 15. 总结

本文档详细拆解了 MineGuard 后端服务的 9 个服务模块，包括：

1. **user-service**：用户服务，负责用户管理、认证授权、司机管理
2. **trip-service**：行程服务，负责行程管理、路线规划、轨迹追踪
3. **dispatch-service**：调度服务，负责任务调度、资源分配
4. **warning-service**：预警服务，负责预警管理、预警检测、预警处理
5. **statistics-service**：统计服务，负责数据统计、报表生成、数据分析
6. **cost-service**：成本服务，负责成本管理、预算管理、薪资管理
7. **ai-service**：AI 分析服务，负责 AI 模型管理、预测分析、智能推荐
8. **python-service**：Python服务，负责Python算法能力
9. **gateway-service**：网关服务，负责请求路由、认证授权、限流熔断

所有服务模块都遵循统一的依赖关系和版本管理策略，通过 HTTP Exchange Client 进行服务间调用，通过 RabbitMQ 进行异步消息通信，通过 WebSocket 进行实时数据推送。

详细的依赖关系和版本管理请参考《后端拆解-依赖关系和版本文档.md》。
