## 3. 服务间调用方式

### 3.1 HTTP Exchange Client（核心机制）

本项目服务间调用采用 **Spring 6 HTTP Exchange** 机制，而非传统的 FeignClient。

**所有 Client 接口定义在 `mineguard-common-core` 模块中**，使用 `@HttpExchange` 注解：

```java
@HttpExchange
public interface UserClient {
    @GetExchange("/user/exists/{id}")
    Result<Boolean> existsUser(@PathVariable("id") Long id);

    @GetExchange("/user/{id}")
    Result<Object> getUserById(@PathVariable("id") Long id);
}
```

| Client 接口 | 说明 | 所在模块 |
| --- | --- | --- |
| UserClient | 用户服务调用 | mineguard-common-core |
| VehicleClient | 车辆服务调用 | mineguard-common-core |
| TripClient | 行程服务调用 | mineguard-common-core |
| DispatchClient | 调度服务调用 | mineguard-common-core |
| WarningClient | 预警服务调用 | mineguard-common-core |
| CostClient | 成本服务调用 | mineguard-common-core |
| AiClient | AI服务调用 | mineguard-common-core |
| StatisticsClient | 统计服务调用 | mineguard-common-core |
| DriverClient | 司机服务调用 | mineguard-common-core |
| MessageClient | 消息服务调用 | mineguard-common-core |

### 3.3 服务依赖关系矩阵

| 服务模块 | 调用的 Client | 被调用 |
| --- | --- | --- |
| user-service | 无 | UserClient, DriverClient |
| vehicle-service | UserClient | VehicleClient |
| trip-service | UserClient, VehicleClient, DispatchClient | TripClient |
| warning-service | UserClient, VehicleClient, TripClient | WarningClient |
| statistics-service | UserClient, VehicleClient, TripClient, WarningClient | StatisticsClient |
| cost-service | UserClient, VehicleClient, TripClient | CostClient |
| ai-service | UserClient, VehicleClient, TripClient, WarningClient, StatisticsClient | AiClient |
| dispatch-service | UserClient, VehicleClient, DriverClient | DispatchClient |
| gateway-service | 无（路由层） | 无 |

**重要说明**：

- 服务间调用通过 Client 接口实现，而非直接依赖其他服务模块
- Client 接口统一放在 `mineguard-common-core` 中，避免循环依赖
- 服务注册发现通过 Nacos 实现，Client 通过服务名调用目标服务

## 5. 版本管理策略

### 5.1 版本号规则

```txt
<major>.<minor>.<patch>
```

- **major**：主版本号，重大功能变更或架构调整时递增
- **minor**：次版本号，新增功能或模块时递增
- **patch**：补丁版本号，修复bug或小的优化时递增

### 5.2 依赖版本管理

1. **父POM集中管理**：所有依赖版本在父POM中统一管理
2. **版本锁定**：使用dependencyManagement锁定版本，避免版本冲突
3. **版本升级**：定期检查和升级依赖版本，确保安全性和兼容性
4. **依赖分析**：使用Maven依赖分析工具，避免依赖冲突和冗余

### 5.3 模块版本管理

1. **公共模块版本**：所有公共模块使用相同版本号，与项目版本保持一致
2. **服务模块版本**：所有服务模块使用相同版本号，与项目版本保持一致
3. **版本发布**：统一发布所有模块，确保版本一致性

## 7. 依赖管理最佳实践

### 7.1 依赖冲突解决

1. **版本锁定**：在父POM中使用dependencyManagement锁定版本
2. **依赖排除**：使用exclusion排除冲突依赖
3. **依赖分析**：使用`mvn dependency:tree`分析依赖树
4. **依赖整理**：定期整理依赖，移除未使用的依赖

### 7.2 依赖安全管理

1. **漏洞扫描**：使用Maven安全插件扫描依赖漏洞
2. **版本更新**：及时更新存在安全漏洞的依赖
3. **依赖审计**：定期审计依赖，确保安全性

### 7.3 依赖性能优化

1. **依赖精简**：只引入必要的依赖，避免过度依赖
2. **依赖版本**：选择稳定且性能好的依赖版本
3. **依赖缓存**：合理配置Maven依赖缓存，提高构建速度

## 8. 构建配置

### 8.1 Maven构建命令

```bash
# 清理构建
mvn clean

# 编译项目
mvn compile

# 运行测试
mvn test

# 打包项目
mvn package

# 安装到本地仓库
mvn install

# 部署到远程仓库
mvn deploy
```

### 8.2 构建插件建议（以父 POM 为准）

构建插件不要在多个子模块重复声明版本；统一在父 POM（`backend/pom.xml`）的 `<build><plugins>` 中管理即可。

```xml
<build>
  <plugins>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-compiler-plugin</artifactId>
      <!-- 版本、source/target 等以父 POM 为准 -->
      <configuration>
        <source>${java.version}</source>
        <target>${java.version}</target>
      </configuration>
    </plugin>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-surefire-plugin</artifactId>
    </plugin>
  </plugins>
</build>
```

## 11. 总结

本文档通过树形结构清晰地展示了后端模块的依赖关系和版本管理策略，实现了：

1. **依赖关系清晰化**：通过树形结构直观展示模块间的依赖关系
2. **版本管理集中化**：在父POM中统一管理所有依赖版本
3. **版本溯源方便化**：详细记录版本历史和规划，便于追溯
4. **依赖冲突最小化**：通过版本锁定和依赖分析，减少依赖冲突
5. **构建配置标准化**：统一构建配置，确保构建一致性
6. **服务调用规范化**：通过 HTTP Exchange Client 实现服务间调用，避免循环依赖

这种依赖管理方式不仅提高了项目的可维护性，也为后续的版本升级和功能扩展奠定了坚实的基础。
