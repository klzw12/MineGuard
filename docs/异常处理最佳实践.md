# MineGuard 异常处理最佳实践

## 1. 异常处理体系架构

### 1.1 核心组件

- **BaseException**：所有自定义异常的基类
- **ExceptionHandlerStrategy**：异常处理策略接口
- **ExceptionHandlerRegistry**：异常处理策略注册中心
- **GlobalExceptionHandler**：全局异常处理器
- **ExceptionHandlerAutoConfiguration**：异常处理自动配置

### 1.2 模块异常类

- **AuthException**：认证授权相关异常（错误码 800-899）
- **DatabaseException**：数据库相关异常（错误码 1000-1099）
- **RedisException**：Redis相关异常（错误码 900-999）
- **WebException**：Web层相关异常（错误码 700-799）

## 2. 错误码分配规范

### 2.1 错误码范围

| 模块 | 错误码范围 | 说明 |
|------|-----------|------|
| common-core | 600-699 | 核心模块通用错误 |
| common-web | 700-799 | Web 层相关错误 |
| common-auth | 800-899 | 认证授权相关错误 |
| common-redis | 900-999 | Redis 缓存相关错误 |
| common-database | 1000-1099 | 数据库相关错误 |
| common-mq | 1100-1199 | 消息队列相关错误 |
| common-log | 1200-1299 | 日志相关错误 |
| common-mongodb | 1300-1399 | MongoDB 相关错误 |
| common-file | 1400-1499 | 文件存储相关错误 |
| common-websocket | 1500-1599 | WebSocket 相关错误 |

### 2.2 错误码设计原则

- **唯一性**：每个错误码在全局范围内唯一
- **可读性**：错误码应具有一定的语义，便于理解
- **扩展性**：预留足够的错误码空间，便于后续扩展
- **一致性**：错误码格式和命名保持一致

## 3. 异常使用示例

### 3.1 抛出异常

**Web模块示例**：

```java
// 参数校验错误
if (StringUtils.isEmpty(username)) {
    throw new WebException(WebResultCode.PARAM_MISSING.getCode(), "用户名不能为空");
}

// 文件上传错误
if (file.getSize() > maxSize) {
    throw new WebException(WebResultCode.FILE_SIZE_EXCEEDED.getCode(), "文件大小超限");
}
```

**Auth模块示例**：

```java
// 权限不足
if (!hasPermission(userId, permission)) {
    throw new AuthException(AuthResultCode.PERMISSION_DENIED.getCode(), "权限不足");
}

// Token过期
if (tokenExpired) {
    throw new AuthException(AuthResultCode.TOKEN_EXPIRED.getCode(), "Token已过期");
}
```

**Database模块示例**：

```java
// 数据源配置错误
try {
    dataSource.setFilters(filters);
} catch (Exception e) {
    throw new DatabaseException(DatabaseResultCode.DATA_SOURCE_ERROR.getCode(), "Druid数据源配置失败", e);
}
```

**Redis模块示例**：

```java
// 限流
if (!redisRateLimiter.tryAcquire(key, limit, window, timeUnit)) {
    throw new RedisException(RedisResultCode.RATE_LIMIT_EXCEEDED.getCode(), "请求频率超限");
}
```

### 3.2 异常处理

**全局异常处理**：

- 所有异常会被 `GlobalExceptionHandler` 捕获
- 根据异常类型自动选择对应的处理策略
- 返回标准化的错误响应

**自定义异常处理**：

```java
@RestControllerAdvice
public class CustomExceptionHandler {
    
    @ExceptionHandler(MyException.class)
    public Result<?> handleMyException(MyException e) {
        return Result.fail(e.getCode(), e.getMessage());
    }
}
```

## 4. 最佳实践

### 4.1 异常抛出原则

1. **具体明确**：抛出具体的异常类型，避免使用通用异常
2. **信息完整**：异常信息应包含足够的上下文信息
3. **异常链**：保留原始异常，便于排查问题
4. **适度使用**：不要过度使用异常，对于可预期的错误应使用返回值

### 4.2 异常处理原则

1. **统一处理**：使用全局异常处理器统一处理异常
2. **分级处理**：根据异常类型和严重程度进行分级处理
3. **日志记录**：记录异常详情，便于问题排查
4. **友好返回**：返回用户友好的错误信息

### 4.3 性能考虑

1. **异常开销**：异常会产生一定的性能开销，避免在高频调用中使用异常
2. **异常粒度**：合理设计异常粒度，避免过于细粒度的异常
3. **异常恢复**：对于可恢复的异常，应提供恢复机制

## 5. 代码规范

### 5.1 异常类命名

- 异常类应以 `Exception` 结尾
- 命名应反映异常的具体类型
- 示例：`AuthException`、`DatabaseException`

### 5.2 错误码枚举命名

- 错误码枚举应以 `ResultCode` 结尾
- 枚举值应使用大写字母和下划线
- 示例：`WebResultCode`、`AuthResultCode`

### 5.3 异常处理策略命名

- 异常处理策略应以 `ExceptionHandlerStrategy` 结尾
- 命名应反映处理的异常类型
- 示例：`WebExceptionHandlerStrategy`、`AuthExceptionHandlerStrategy`
