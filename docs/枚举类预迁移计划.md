# 枚举类预迁移计划

## 背景

当前所有业务枚举类都存放在 `mineguard-common-core` 模块中，这是因为具体的服务模块（user-service、vehicle-service等）尚未建立。当服务模块建立后，这些枚举类应该迁移到对应的业务模块中，以遵循高内聚低耦合的设计原则。

## 当前状态

### 已存在的枚举类（按模块分布）

| 模块 | 枚举类 | 数量 |
|------|--------|------|
| mineguard-common-core | 22个业务枚举类 | 22 |
| mineguard-common-auth | RoleEnum, PermissionTypeEnum | 2 |
| mineguard-common-file | FileBusinessTypeEnum, FileRecordStatusEnum | 2 |
| mineguard-common-websocket | MessageStatusEnum, MessageTypeEnum, MessagePriority | 3 |

**问题**：core模块承载了过多业务枚举类，违反了模块职责单一原则。

## 迁移实施建议

### 阶段一：服务模块建立时
1. 创建服务模块的基础结构
2. 将对应的枚举类从core模块复制到新模块
3. 保持原有包名结构（`com.klzw.xxx.enums`）

### 阶段二：代码迁移时
1. 逐步替换引用，从 `com.klzw.common.core.enums.XxxEnum` 改为 `com.klzw.xxx.enums.XxxEnum`
2. 确保所有引用都更新后再删除core模块中的旧枚举类

### 阶段三：清理阶段
1. 删除core模块中已迁移的枚举类
2. 更新文档
3. 运行全量测试确保无遗漏

## 注意事项

1. **兼容性**：迁移过程中保持枚举值不变，避免数据不一致
2. **版本控制**：每次迁移一个模块，便于回滚
3. **文档同步**：及时更新接口文档和数据库文档
4. **测试覆盖**：迁移后需要全量测试相关业务功能

## 当前临时方案

在服务模块建立之前，暂时将所有枚举类放在core模块是可以接受的。但应该在代码注释中明确标注将来需要迁移的目标模块，例如：

```java
/**
 * 司机状态枚举
 * 对应db.sql中driver表的status字段
 * TODO: 将来迁移到 user-service 模块
 */
@Getter
public enum DriverStatusEnum {
    // ...
}
```
